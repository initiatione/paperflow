# Paper Source / Paper Wiki Integration

Paper Wiki (machine-facing name `paper-wiki`; `prw` is a pre-Stage-2 legacy alias) consumes Paper Source (machine-facing name `paper-source`; `epi` is a pre-Stage-2 legacy alias) handoff artifacts and does not replace Paper Source.

Paper Wiki owns the wiki-side closed loop: read vault state, consume Paper Source handoff artifacts, write or repair formal pages, maintain formal page content relationships, detect claim staleness and evidence-tier drift, split or merge pages when reusable knowledge changes, update tracking files, refresh `final-source-review.json`, run post-task checks, and report whether the paper is ready for Paper Source recording. `prw` is only a legacy alias.

Paper Source owns paper discovery, ranking, download, MinerU parsing, paper wiki vault bootstrap through Paper Source `wiki-setup`, `paper-gate`, human approval, and `record-wiki-ingest`. `epi` is only a legacy alias.
Paper Source owns human approval records and `record-wiki-ingest`; Paper Wiki reports readiness but does not replace those records.

Paper Wiki assumes an initialized vault contract. It may check for the core `_paper_source` bootstrap (`_paper_source/`, `_paper_source/raw/`, `_paper_source/staging/`, `_paper_source/meta/`, `_paper_source/policies/`), legacy `_epi/` bootstrap, `_meta/`, `.obsidian`, `.git`, and the seven formal page roots, but it does not initialize the vault, does not reset the vault, and does not silently create missing vault structure. If missing core vault structure blocks paper wiki work, report the capability gap and point the user back to Paper Source `wiki-setup`. Missing `_paper_source/runs/`, `_paper_source/cache/`, `_paper_source/tmp/`, `_paper_source/tmp-manual-pdfs/`, `_paper_source/quarantine/`, or `_paper_source/evolution/` is not a bootstrap failure; Paper Source creates those directories on demand.

The canonical Paper Source-to-Paper Wiki handoff is `wiki-ingest-brief.json`. Required inputs include the brief, reading report, metadata, PDF, MinerU Markdown, images, manifest, `figure-index.json`, `formula-index.json`, and `asset-normalization-record.json`. MinerU Markdown is the primary source for formulas, notation, method context, and prose; PDF, figure/formula indexes, and image evidence are fallback checks when Markdown is missing, wrong, ambiguous, or insufficient. A non-empty native `mineru/paper.tex` may be read only as an optional cross-check when present. `wiki_deposition_task.json` is a legacy compatibility artifact; Paper Wiki may read it for historical context, but task-only legacy handoffs are not ready for formal writes until Paper Source regenerates or repairs the brief. This canonical handoff / legacy compatibility split applies to new Paper Wiki intake.

Paper Source owns raw MinerU asset normalization. Its new parse and `normalize-mineru-assets --execute` may rename hash-like raw images, remove Markdown formula screenshot embeds, and write figure/formula indexes. Paper Wiki consumes those indexes for formal page maintenance, but it must not rename files under `_paper_source/raw/**` or legacy `_epi/raw/**`.

Paper Wiki figure maintenance is a formal-page workflow. When formal pages still embed hash image paths, Paper Wiki should audit the formal page roots, resolve old paths through `figure-index.json` and `formula-index.json`, snapshot pages before material repairs, patch only evidence image/formula/provenance lines, and run the normal post-task check. If indexes are missing, report that Paper Source raw normalization must run first instead of repairing raw assets from Paper Wiki.

If source artifacts are missing, stop and send the user back to Paper Source `paper-gate`. Human approval remains a Paper Source record; this plugin must not write human approval state. After formal pages are written and `final-source-review.json` exists, Paper Source `record-wiki-ingest` records final paths and hashes.

Paper Wiki may read previous `wiki-ingest-record.json` files to find affected formal pages, hashes, and reverse dependencies, but it must not refresh or replace those Paper Source record files. Paper Wiki records readiness; Paper Source writes or replaces `wiki-ingest-record.json` through `record-wiki-ingest`.

The read-only `ask_wiki` route is different from record-request ask-mode automation. `ask-wiki` answers user questions from the formal Obsidian graph, labels evidence/synthesis/inference/uncertainty, reports correction candidates, and asks before repair. It is the conversational primary entrypoint; Paper Source `wiki-ask` CLI is the same-source fallback / 程序化 `--json` entry. It does not write `log.md`, does not write formal pages, does not refresh QMD, does not write Paper Source artifacts, and ask-wiki does not write `paper-wiki-record-request.json`.

Ask-mode automation uses `_paper_source/staging/papers/<paper-slug>/paper-wiki-record-request.json`. Paper Wiki writes the request artifact; Paper Source consumes it. The request uses `schema_version: paper-wiki-record-request-v1`, `automation_mode: ask`, final page paths and sha256 hashes, `final-source-review.json` path/hash, and the human approval identity. The next Paper Source command is `record-wiki-ingest --from-paper-wiki-request _paper_source/staging/papers/<paper-slug>/paper-wiki-record-request.json`; Paper Source validates the live vault again before writing or replacing `wiki-ingest-record.json`. Legacy `_epi/staging/papers/<paper-slug>/prw-record-request.json`, `schema_version: prw-record-request-v1`, and `--from-prw-request` remain accepted for existing artifacts only.
