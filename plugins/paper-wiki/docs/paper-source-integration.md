# Paper Source / Paper Wiki Integration

Paper Wiki (machine-facing name `paper-wiki`) consumes Paper Source (machine-facing name `paper-source`) handoff artifacts and does not replace Paper Source. Historical aliases are not Paper Wiki user entrypoints, route triggers, or new artifact contracts.

Paper Wiki owns the wiki-side closed loop: read vault state, consume Paper Source handoff artifacts, write or repair formal pages, maintain formal page content relationships, detect claim staleness and evidence-tier drift, split or merge pages when reusable knowledge changes, update tracking files, refresh `_meta/reference-index.json`, refresh `final-source-review.json`, run post-task checks, and report whether the paper is ready for Paper Source recording.

Paper Source owns paper discovery, ranking, download, MinerU parsing, paper wiki vault bootstrap through Paper Source `wiki-setup`, paper-level deduplication, `paper-gate`, human approval, and `record-wiki-ingest`.
Paper Source owns human approval records and `record-wiki-ingest`; Paper Wiki reports readiness but does not replace those records.

Paper Wiki assumes an initialized vault contract. It may check for the core `_paper_source` bootstrap (`_paper_source/`, `_paper_source/raw/`, `_paper_source/staging/`, `_paper_source/meta/`, `_paper_source/policies/`), `_meta/`, `.obsidian`, `.git`, and the seven formal page roots, but it does not initialize the vault, does not reset the vault, and does not silently create missing vault structure. If missing core vault structure blocks paper wiki work, report the capability gap and point the user back to Paper Source `wiki-setup`. Missing `_paper_source/runs/`, `_paper_source/cache/`, `_paper_source/tmp/`, `_paper_source/tmp-manual-pdfs/`, `_paper_source/quarantine/`, or `_paper_source/evolution/` is not a bootstrap failure because Paper Source creates on-demand directories when needed.

The canonical Paper Source-to-Paper Wiki handoff is `wiki-ingest-brief.json`. Required inputs include the brief, reading report, metadata, PDF, MinerU Markdown, images, manifest, `figure-index.json`, `formula-index.json`, and `asset-normalization-record.json`. MinerU Markdown is the primary source for formulas, notation, method context, and prose; PDF, figure/formula indexes, and image evidence are fallback checks when Markdown is missing, wrong, ambiguous, or insufficient. A non-empty native `mineru/paper.tex` may be read only as an optional cross-check when present. Paper Wiki treats Paper Source handoffs as paper-level deduplicated; it does not run paper deduplication and instead performs graph-level merge-before-create against existing formal concepts, derivations, experiments, synthesis, reports, and opportunities.

After Paper Wiki writes or rewrites a `references/` page, it must refresh `_meta/reference-index.json` with the changed page and any raw-only source metadata. This index is the canonical lightweight backlog that Paper Source discovery uses for DOI/arXiv/source_id/title dedupe; Paper Source should not have to rescan formal pages or infer deposited status from chat history.

Paper Source owns raw MinerU asset normalization. Its new parse and `normalize-mineru-assets --execute` may rename hash-like raw images, remove Markdown formula screenshot embeds, and write figure/formula indexes. Paper Wiki consumes those indexes for formal page maintenance, but it must not rename files under `_paper_source/raw/**`.

Paper Wiki figure maintenance is a formal-page workflow. When formal pages still embed hash image paths, Paper Wiki should audit the formal page roots, resolve old paths through `figure-index.json` and `formula-index.json`, snapshot pages before material repairs, patch only evidence image/formula/provenance lines, and run the normal post-task check. If indexes are missing, report that Paper Source raw normalization must run first instead of repairing raw assets from Paper Wiki.

If source artifacts are missing, stop and send the user back to Paper Source `paper-gate`. Human approval remains a Paper Source record; this plugin must not write human approval state. After formal pages are written and `final-source-review.json` exists, Paper Source `record-wiki-ingest` records final paths and hashes.

Ordinary approval-waiting handoff means Paper Wiki can read `wiki-ingest-brief.json`, report that Paper Source approval is still required, and stop before formal page writes. Explicit Codex automation is different: Paper Wiki may be driven by the running Codex task after Paper Source has written an auditable `wiki-agent-trigger.json` with `automation_handoff` and `approved_by=codex-automation:<task-id>`. Paper Wiki treats that trigger as permission context only; it does not write `human-approval.json`, does not write `wiki-agent-trigger.json`, and does not write or replace `wiki-ingest-record.json`.

Paper Wiki may read previous `wiki-ingest-record.json` files to find affected formal pages, hashes, and reverse dependencies, but it must not refresh or replace those Paper Source record files. Paper Wiki records readiness; Paper Source writes or replaces `wiki-ingest-record.json` through `record-wiki-ingest`.

The read-only `ask_wiki` route is different from record-request ask-mode automation. `ask-wiki` answers user questions from the formal Obsidian graph, labels evidence/synthesis/inference/uncertainty, reports correction candidates, and asks before repair. It is the conversational primary entrypoint; Paper Source `wiki-ask` CLI is the same-source fallback / 程序化 `--json` entry. It does not write `log.md`, does not write formal pages, does not refresh QMD, does not write Paper Source artifacts, and ask-wiki does not write `paper-wiki-record-request.json`.

Ask-mode automation uses `_paper_source/staging/papers/<paper-slug>/paper-wiki-record-request.json`. Paper Wiki writes the request artifact; Paper Source consumes it. The request uses `schema_version: paper-wiki-record-request-v1`, `automation_mode: ask`, final page paths and sha256 hashes, `final-source-review.json` path/hash, and the human approval identity. The next Paper Source command is `record-wiki-ingest --from-paper-wiki-request _paper_source/staging/papers/<paper-slug>/paper-wiki-record-request.json`; Paper Source validates the live vault again before writing or replacing `wiki-ingest-record.json`.
