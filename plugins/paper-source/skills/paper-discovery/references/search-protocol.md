# Search Protocol

Treat `paper_search_mcp` as a search transport, not as the definition of quality. The user usually wants papers that are worth reading, not just papers that one source happened to return. Borrow the discipline of multi-source academic search: explicit query construction, source routing, deduplication, citation/venue verification, and a visible evidence trail.

Before running the first query:

1. Build a query plan using `query-planner.md` and the optional `scripts/query-planner.py` helper.
2. Split the request using the user's config/profile: discipline/domain, method or topic terms, problem/task, context, quality evidence, venue prior, and exclusions.
3. Build 5-8 query variants instead of one vague query. Include exact phrases, synonyms, acronym expansions, and task-specific terms derived from config first; load `domain-ontology.md` only as optional examples. For natural-language or Chinese/English mixed requests, this compilation is agent work, not Python hardcoding.
4. Convert explicit request constraints into executable fields: use `year_min` for a stated recent window and `code_policy=prefer` or `code_policy=require` for public-code requests. `prefer` is a ranking preference; `require` is a hard metadata gate.
5. Default `discover-papers` discovery keeps review/survey/meta candidates for classification and ranking. Add `-review -survey` query exclusions and enforce review/survey/meta filtering only when the user explicitly asks for non-review/non-survey papers.
6. Route sources by intent: use `paper_search_mcp` first with configured broad keyword sources such as `arxiv,semantic,openalex,crossref`. Pass compiled queries via repeated `--query-variant` or a structured `--agent-query-plan-json`; keep the raw `--query` as the intent label and evidence signature, not as the primary MCP query. Unpaywall is a DOI lookup / OA locator, so it belongs in exact DOI lookup and acquisition fallback, not no-DOI first-pass keyword search. Optional `grok-search-rs` MCP is a supplemental source for recall gaps, not a replacement for `paper_search_mcp`. Live academic/web search is a targeted verification fallback only after Paper Source has a query plan, provider gap signal, candidate identity, or run artifact (`search-record.json`, `report.json`, `rank.json`, a DOI/arXiv/title from the user, or a repository URL found in a source bundle). Do not use Firecrawl, generic web search, publisher search, or GitHub search as the primary discovery provider for paper recommendations.
7. Apply `two-stage-retrieval.md`: first form a high-recall candidate pool, then deduplicate, verify, and precision-rank.
8. Apply the `hard_domain_anchors` gate before ranking only when the query plan provides confirmed anchors from user/config/Research Brief/agent hard constraints. A paper that only matches a broad method term, such as reinforcement learning, graph neural network, or deep learning, should not be rejected by inferred n-grams alone; keep inferred terms in `soft_recall_terms` and let ranking/diagnostics explain priority.
9. Apply request constraints before ranking. Candidates older than `year_min` are rejected as `year_before:<year>`; missing years are rejected as `year_missing` when a recency gate is explicit. `code_policy=require` rejects missing `code_url` / repository identity as `missing_code`; `code_policy=prefer` keeps the paper but raises reproducibility/ranking signals for code-bearing candidates.
10. Apply `venue-prior.md` as a recall/ranking prior from user config. Check whether the user's configured flagship venues, journals, conferences, or field databases are missing before accepting the result set.
11. Deduplicate across variants by DOI first, arXiv base id second, then source id and normalized title.
12. Deduplicate against the wiki backlog in `_meta/reference-index.json` before ranking. A paper already deposited as a formal reference page must be rejected with `already_in_wiki:<page>` and should not be recommended again unless the user explicitly asks to update or revisit it.
13. Fall back to the downloaded source library under `_paper_source\\raw`. A paper already present in `_paper_source\\raw\\<slug>\metadata.json` must be rejected with `already_in_library:<slug>` and should not be recommended again unless the user explicitly asks to repair or reprocess that existing paper.
14. For strong seed papers, use `citation-graph.md` to check journal versions, related papers, references, and recent cited-by papers.

After each `discover-papers` run, inspect the linked discovery run's `report.json.discovery_context.source_coverage` or the `Source Coverage` / `Source Routing` sections in `report.md`. For explicit evidence/debug runs, inspect the same fields directly after `dry-run`. Treat `sources_used`, `source_results`, `errors`, `raw_total`, `deduped_total`, `query_count`, `capabilities`, `provider_readiness`, `source_routing`, and `provider_gaps` as recall evidence: a zero-count or errored configured source is a concrete gap to rerun or explain, an unsupported download/read capability is a source boundary, and a missing provider env such as `unpaywall_email_missing` is a recall/fallback risk rather than cosmetic telemetry. Google Scholar is unstable by default and should stay demoted unless the user asks for it or a clear recall gap requires a targeted rerun. Use `doctor --json` to inspect `paper_search_provider_readiness` before blaming ranking when OA fallback is weak.

## Optional Grok Supplemental Discovery

`grok-search-rs` MCP can be configured as an optional supplemental discovery provider through runtime config (`grok_search_mcp.command`, `args`, and env files). Keep API keys in env files or process environment; do not write secrets into skill docs, vault config, runtime status, reports, or run artifacts.

Default Grok mode is `targeted` when the runtime command is configured; otherwise discovery silently resolves to `off` and records sanitized `not_configured` diagnostics. In `targeted`, Paper Source runs `paper_search_mcp` first and skips Grok when paper-search returns at least 3 Tier A/B or `advance-candidate` results, at least 2 staging-ready or stable-identity results, and no required/recommended provider gap. When it does run, targeted Grok uses 3-10 supplemental queries, default 5, prioritized by provider/query gaps, academic publisher/digital-library gaps, candidate identity/PDF gaps, then broad recall gaps.

Use `--grok-mode parallel` only when the user wants more aggressive recall. Parallel mode is "parallel shortfall search": Grok starts alongside the full paper-search lifecycle with a static query set derived from the initial query plan, source routing, and gap domains. It should cover paper-search query variants where possible and then spend remaining budget on gap-domain recall such as IEEE, ACM, ScienceDirect, Springer, JSTOR, Web of Science, Scopus, and ResearchGate. It must not stop, shorten, or reroute paper-search. After paper-search finishes with usable candidates, wait only the fan-in grace for Grok; if paper-search fails or returns no usable candidates, Grok may finish as salvage evidence, but Grok-only salvage is diagnostics/evidence and not a final recommendation.

Final recommendations need at least one usable paper-search candidate. A usable paper-search candidate does not require a PDF; stable DOI/arXiv/publisher landing/Semantic/OpenAlex/CrossRef-like identity plus non-rejected ranking/readiness is enough. Grok-only stable paper candidates may appear only as supplemental recommendations when the same final recommendation set includes paper-search usable candidates, capped by `grok_only_recommendation_cap` and never above 5. Duplicate papers use paper-search-priority best-field merge: DOI/arXiv/title/authors/year/venue/abstract stay from paper-search, while Grok may add PDF URLs, publisher landing URLs, alternate sources, and evidence.

Before any repeated search for the same topic, check `_meta/reference-index.json` for deposited papers, then inspect whether a matching `_paper_source/reviews/<review-id>/` provider-cache session or recent `_paper_source/runs/<run-id>/search-record.json` already exists. Default to `discover-papers` / `dry-run` resume with wiki-index dedupe, `report --run-id`, or direct artifact inspection. Use `--refresh` only when the user asks for freshness, provider gaps require a rerun, or the previous run is demonstrably stale for the current task.

For code-bearing papers, verify code availability only after paper identity is anchored by Paper Source or by a user-provided DOI/arXiv/title. Treat repository checks as code evidence verification, not as paper discovery. Record the checked URL, repository identity, command or page used, and static-vs-run boundary in the source evidence surface before Paper Wiki reuses it.

For any narrow field, do not let the query planner blur the target into generic AI/science terms. If the first planned run broadens away from the user's configured profile or current request, rerun with `--no-query-plan` and exact phrases from the user's wording, or update config before rerunning.

If the current request needs a hard domain gate, the agent must pass explicit anchors with `--domain-focus-term`; keep the hard anchor gate on those terms and the user's configured domain terms. Long-query n-grams that describe only a broad task or object class are recall aids, not sufficient evidence for domain fit.

Treat publisher PDF blocks as acquisition evidence, not as the end of discovery. HTTP 403/502 from IEEE, Wiley, or publisher PDFs should be reported as publisher PDF blocks; then try open sources, Unpaywall, arXiv versions, DOI landing pages, Semantic Scholar/OpenAlex metadata, or a sharper arXiv-first query before concluding that the paper cannot be processed.

During acquisition, Paper Source should pass stable DOI/title/source identity to paper-search MCP `download_with_fallback` first. The default fallback chain is source-native, OpenAIRE, CORE, Europe PMC, PMC, and Unpaywall; Sci-Hub is disabled unless the user explicitly opts in. If a candidate has no direct PDF URL and OA fallback produces no PDF, Paper Source should stop quickly with `failure_class=manual-download-required` and `manual_download.candidate_manual_urls` so the user can use the DOI/publisher links and download through their organization/institution. These links must also surface through prepare-ranked report `manual_downloads` and `runs-query --json` so the conversation can give the user direct DOI/publisher jump links without digging through raw JSON. If a direct PDF URL exists, Paper Source may still try source-native MCP, CLI, and direct URL recovery. A DOI or publisher landing page is not a PDF. If a landing page exposes `citation_pdf_url` or an obvious publisher PDF link, Paper Source can follow it once. When merged candidates provide multiple `pdf_urls`, acquisition should try each candidate URL in order and record `candidate_pdf_urls` plus `acquire_attempts`; only after those fallbacks fail should `acquire-record.json` report `failure_class=not-pdf`, `manual-download-required`, or a publisher/network block and ask for a direct PDF/open-access source rather than sending HTML to MinerU. If the downloaded PDF's DOI/title clearly mismatches the candidate, Paper Source writes `identity-check.json`, reports `failure_class=identity-mismatch`, quarantines the PDF under `_paper_source/quarantine/papers/<slug>/paper.pdf`, and must not send it to MinerU.

If the first result set is generic, stale, too review-heavy, or full of blocked PDFs, run a sharper rerun rather than stopping early.
